# Milestone v1.1: Explainability + Feedback Loop Closure

**Status:** ✅ SHIPPED 2026-05-19 (archived retroactively 2026-06-20)
**Phases:** 1–9
**Total Plans:** 26

## Overview

Took the recommendation engine from "works but opaque" to explainable and self-improving. Fixed the broken foundation (known-song filtering, crashing personalization engine, dead test suite), built a real content-based scorer (genre taste vector + cosine similarity + novelty), closed the online feedback loop (taste-vector updates + Thompson-sampling bandit over candidate sources), hardened security for production, then persisted score breakdowns to the DB, surfaced them in the UI, and documented the whole system for interviews.

## Phases

### Phase 1: Fix Foundation (4 plans)
DB-backed known-song exclusion (RecommendationLog + DailyGem history); fixed `Count` import + `update_weights` arity crash; `RecommendationLog.liked` written on like/dislike/unlike; `DailyGem.was_liked` synced; track-level exclusion; `artist_related_artists` 5th strategy; test suite repaired (31 tests pass).

### Phase 3: Feedback Learning Loop (4 plans)

### Phase 4: Metrics, Evaluation & Documentation (4 plans)

### Phase 6: Schema Migration (2 plans)
`DailyGem` gains `score_breakdown` (JSON), `score_total` (float), `was_saved` (bool), `taste_vector_snapshot` (JSON); migration 0008; 113 tests pass.

## Milestone Summary

**Key Decisions:**

- Content-based filtering as ML approach — no multi-user base; cosine similarity on a user preference vector is explainable and implementable.
- Drop TF-IDF for taste vector — frequency count sufficient at portfolio scale.
- Remove audio-weight dead code — adjusted deprecated Spotify audio features.
- Compound metric `liked OR saved` (not AND) — AND too sparse for single-user daily-gem app.
- Deterministic explanation template (no OpenAI) — zero cost, zero latency, formula-synchronized by construction.
- `score_breakdown` as JSONField + `score_total` flat FloatField — one migration, open schema, queryable aggregate.
- `taste_vector_snapshot` at recommendation time — captures model state at pick time for offline evaluation.

**Issues Resolved:**

- Recommendations included already-known songs (Phase 1).
- Static source weights, no ML scoring (Phase 2).
- `RecommendationLog.liked` never updated (Phase 1).
- Personalization engine crash on use (Phase 1).
- Score breakdown silently `{}` on cached responses — fixed all 3 cached-branch return sites (Phase 7).

**Issues Deferred (acknowledged at v1.2 close):**

- Phase 03 UAT: 2 pending scenarios (live score_breakdown flow; Thompson bandit convergence) — require running app.
- Phase 05 UAT: 4 pending scenarios (CSRF round-trip/rejection, frontend env vars, OAuth) — code clean (9/9 automated), browser-only.
- Phase 08 UAT: 4 pending scenarios (bars render, empty state, hit-rate tile, null fallback) — browser-only.
- Phase 03 VERIFICATION (human_needed) — see Technical Debt.
- Phase 05 VERIFICATION (human_needed) — browser-only, no code gap.

**Technical Debt Incurred:**

Was a ROADMAP Phase 3 deliverable, never specified as a plan must-have.
