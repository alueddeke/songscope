## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| test code → production schema | Test assertions encode contract expectations about field defaults and nullability. A weak or missing assertion means the migration could silently regress without test detection in later phases. |
| pytest test DB → migration chain | pytest-django applies all migrations including 0008 against a fresh test SQLite DB. A broken migration would surface here before reaching prod. |

| T-06-08 | Repudiation | Regression baseline | mitigate | Task 2 requires running the full suite and reporting any failure with exact test name; pre-existing failures must be documented in the summary so future phases have a clear baseline |
| T-06-09 | Denial of Service | Test DB migration | mitigate | All four new fields are nullable or have safe defaults, so migration on the test DB cannot fail due to NOT NULL violations — same property as the prod path (T-06-02) |
| T-06-10 | Information Disclosure | Test fixture data | accept | Test users (`newfieldsuser`) and tracks (`'C' * 22` spotify_id) are synthetic; no real PII or credentials enter the test DB |
| T-06-11 | Spoofing | Test isolation | accept | Django TestCase wraps each test in a transaction that rolls back at teardown; setUp creates fresh User/Track/DailyGem per test, preventing cross-test state leakage |
