<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Authenticated client → submit_feedback view | Authenticated user submits feedback_type and track_id from frontend; user-controllable input crosses here |
| submit_feedback → Django ORM | View writes RecommendationLog.liked + DailyGem.was_liked; ORM mediates DB access |
| PersonalizationEngine → UserPreferences ORM | Engine reads/writes user-scoped preference rows |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | T (Tampering) | submit_feedback writing RecommendationLog.liked for arbitrary user | mitigate | ORM filter scoped with `user=request.user` — request.user comes from `@permission_classes([IsAuthenticated])` enforced by DRF; user cannot tamper to update another user's logs |
| T-02-02 | I (Information Disclosure) | feedback_type input from request body | mitigate | The `feedback_type == 'LIKE'` comparison is exact-string; non-LIKE/DISLIKE values fall through to `liked=False` in the LIKE branch — guard with explicit ternary that only writes True for LIKE, False for DISLIKE, else None (handled by the unlike branch) |
| T-02-03 | E (Elevation of Privilege) | apply_feedback_learning being a no-op | accept | No-op cannot grant privilege; the prior crash bug was a DoS vector, the no-op resolves that |
| T-02-04 | D (DoS) | Unbounded RecommendationLog query | mitigate | `.order_by('-recommended_at').first()` returns at most one row; ORM filter is on indexed FKs (user, track) — bounded query |
| T-02-05 | R (Repudiation) | RecommendationLog.liked changes lack audit trail | accept | Phase 1 is restoring the missing write; full audit log is out of scope for this milestone (deferred to ML engine v2) |
| T-02-06 | S (Spoofing) | track_id input | mitigate | Existing submit_feedback already does `track = Track.objects.get(spotify_id=spotify_id)` — invalid IDs raise DoesNotExist and hit the outer except (lines 663-665), returning HTTP 500 without exposing the user's track set |
</threat_model>
