<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| OS filesystem → Django process | `backend/.env` provides `SECRET_KEY`, read at process startup via decouple |
| git history → present working tree | Old SECRET_KEY exists in commits 96d76dcd, d23b44d6 — must NOT be reused |
| browser → Django POST/PUT/PATCH/DELETE endpoints | CSRF token must accompany state-mutating requests |
| Django view code → SessionAuthentication subclass | Removing the bypass class eliminates a future-accidental-use risk |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-01 | Information Disclosure | `SECRET_KEY` in `settings.py` source | mitigate | Replace `os.environ.get('SECRET_KEY', 'django-insecure-fallback-dev-only')` with `config('SECRET_KEY')` — no default. Store new key only in `backend/.env` (which is `.gitignore`d). Generate a fresh key via Django's `get_random_secret_key()` so the historical key in git history is irrelevant. |
| T-05-02 | Spoofing / Tampering | Django state-mutating endpoints (POST /api/submit-feedback/, POST /api/submit-ai-feedback/, etc.) | mitigate | Uncomment `"django.middleware.csrf.CsrfViewMiddleware"` in `MIDDLEWARE` so Django enforces CSRF tokens on all unsafe methods. The frontend CsrfProvider + axios `X-CSRFToken` header are already wired (per RESEARCH §Issue 3) so the round-trip works. |
| T-05-03 | Elevation of Privilege | `CsrfExemptSessionAuthentication` class in views.py | mitigate | Delete the class definition and its `SessionAuthentication` import. Currently unused (grep confirmed no view sets `authentication_classes = [CsrfExemptSessionAuthentication]`), but leaving dead-code bypasses around is a defence-in-depth hazard. |
| T-05-04 | Denial of Service | Django startup with missing `SECRET_KEY` | accept | python-decouple raises `UndefinedValueError` at startup if `SECRET_KEY` is absent — this is intentional fail-loud behavior, preferable to silent fallback to a known weak key. Documented in `.env.example` (created in Plan 03). |
| T-05-05 | Repudiation | Existing test suite stability | accept | Django TestClient sets `enforce_csrf_checks=False` by default; all 77 existing tests use `self.client.force_login` + POST which bypasses CSRF. Grep confirmed no test uses `enforce_csrf_checks=True`. Risk of test breakage is very low; full suite run is the gate. |
</threat_model>
