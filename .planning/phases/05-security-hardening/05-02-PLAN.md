<objective>
Remove Spotify credentials from the Next.js frontend build pipeline.

Purpose: Eliminate security issue SEC-02 — `SPOTIFY_CLIENT_SECRET` and `SPOTIFY_CLIENT_ID` are currently exposed to the browser JavaScript bundle via `next.config.mjs`'s `env` block. The credentials are NEVER consumed by frontend code (verified by grep in RESEARCH.md §Issue 2). The exposure is pure liability. The OAuth flow is entirely Django-side; the frontend only needs `NEXT_PUBLIC_BACKEND_URL`, which Next.js reads automatically from `.env.local`.

Output: Simplified `frontend/next.config.mjs` (no env block, no dotenv import), optional `dotenv` removal from `package.json`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser ← Next.js JS bundle | Anything baked into client JS at build time is publicly readable in DevTools |
| Build pipeline → frontend bundle | `env:` block in `next.config.mjs` inlines values into client-side code |
| Frontend ↔ Django backend | OAuth flow is server-side; frontend only needs `NEXT_PUBLIC_BACKEND_URL` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-02 | Information Disclosure | `SPOTIFY_CLIENT_SECRET` via `next.config.mjs` `env` block | mitigate | Delete the entire `env` block from `next.config.mjs`. The credential is never consumed in any frontend file (grep-verified) so removal is zero-impact. After this, even a deployed bundle cannot leak the secret. |
| T-05-06 | Information Disclosure | `SPOTIFY_CLIENT_ID` exposed in the same `env` block | mitigate | Same fix — removed in the same deletion. The CLIENT_ID alone is less sensitive but exposing it has no upside since the frontend doesn't use it. |
| T-05-07 | Tampering | `REDIRECT_URI` exposed in the `env` block | mitigate | Same fix — removed in the same deletion. The redirect URI is configured server-side in Django (`SPOTIFY_REDIRECT_URI` via settings.py) and frontend code does not need it. |
| T-05-08 | Denial of Service | Frontend build failure due to missing env vars after change | accept | The only frontend-consumed var is `NEXT_PUBLIC_BACKEND_URL`, which Next.js resolves automatically from `frontend/.env.local`. Build is verified by `npm run build` exit-0 acceptance check. |
</threat_model>
